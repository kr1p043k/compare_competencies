## 1. Project Overview

Python/FastAPI + React + PostgreSQL project for comparing educational competencies with market demands. Direction: 09.03.02 "Перспективные информационные технологии" (SFEDU).

**Pipeline flow:** РПД PDFs → RPDLoader (parse) → RPDSkillCleaner → seed_db (load to PostgreSQL) → teacher_analysis_runner (CoverageAnalyzer + CurriculumRecommender) → JSON results + DB coverage_analyses + charts.

## 4. RPD → DB Pipeline (How Skills Get Loaded)

### Stage A: PDF Parsing (`rpd_loader.py`)
- Scans `temp/rpd_pdfs/` for РПД PDFs
- Extracts text via pypdf (EasyOCR fallback for low-Cyrillic)
- Parses competency codes: ОПК-\d+, ПК-\d+, УК-\d+, ППК-[РС]\d+, ИП-\d+
- Parses KSA sections: "Знания:", "Умения:", "Навыки:" (Russian + English)
- Merges child competencies into parents (ОПК-2.1 → ОПК-2)
- Deduplicates English+Russian PDF variants
- Output: `data/reference/krm_disciplines_09.03.02.json`

### Stage B: Skill Cleaning (`rpd_skill_cleaner.py`)
- Filters noise: exam scheduling, evaluation criteria, verbose prose
- Normalizes: strips prefixes, lowercases
- Output: cleaned `data/reference/rpd_skills.json`

### Stage C: DB Seeding (`seed_db.py`)
1. `seed_skills()` — loads `it_skills.json` (933 skills) + `rpd_skills.json` → `skills` table
2. `seed_krm()` — reads `krm_disciplines_09.03.02.json` → creates Direction, Discipline, Competency, KSAEntry, CompetencySkill records

### Stage D: DB Repair (`fix_rpd_data.py`)
5-step repair: clean noisy rpd_skills → ensure clean taxonomy → populate ksa_entries → rematch competencies to it_skills

## 6. Issues Found

### BUG 1: `analysis_type="teacher"` vs CHECK constraint `"teacher-analysis"`
**File:** `src/pipeline/teacher_analysis_runner.py:766`
```python
await save_to_analysis_results(run_id=run_id, analysis_type="teacher", data=dir_summary)
```
The DB CHECK constraint on `analysis_results.analysis_type` requires one of: `('gap', 'coverage', 'cluster', 'trend', 'teacher-analysis')`. Value `"teacher"` will violate the constraint.

**Fix:** Change `"teacher"` → `"teacher-analysis"`.

### BUG 2: `competency_tree` endpoint references non-existent column
**File:** `src/api_pkg/routers/teacher.py:430-436`
```python
cov_subq = select(
    CoverageAnalysis.discipline_id,
    CoverageAnalysis.competency_id,  # ← THIS COLUMN DOESN'T EXIST
    func.avg(CoverageAnalysis.coverage_ratio).label("weighted_coverage"),
)
```
`CoverageAnalysis` model has `discipline_id`, `direction_id`, `total_skills`, `market_matched_skills`, `coverage_ratio`, `analysis_date` — but NO `competency_id` column. The SQL schema confirms this.

**Fix:** Either add `competency_id` FK to `coverage_analyses` table, or rewrite the query to aggregate by discipline only.

### BUG 3: `_is_rpd()` doesn't filter "ГИА" files
**File:** `src/loaders/rpd_loader.py:572-574`
```python
def _is_rpd(self, fname: str) -> bool:
    skip_keywords = ["Rating", "ECTS", "BPM", "Аннотация", ".sig"]
    return not any(kw in fname for kw in skip_keywords)
```
`ГИА_ИИ 09.03.02.pdf` will NOT be filtered out (no "ГИА" in skip_keywords). The parser will attempt to extract competencies from a ГИА document, which has a different structure.

**Fix:** Add "ГИА" to skip_keywords.

### ISSUE 4: `teacher_stats` reads from file, not DB
**File:** `src/api_pkg/routers/teacher.py:59`
The `/teacher/stats` endpoint reads from `krm_disciplines_09.03.02.json` file instead of querying the database. If the file is missing (which it currently is), stats return 0.

**Fix:** Query DB instead of reading file, or handle the missing file case more gracefully.

### ISSUE 5: `krm_market_skills` returns frequency=1 for all skills
**File:** `src/api_pkg/routers/teacher.py:273`
```python
return [{"skill": s, "frequency": 1} for s in list(skills)[:limit]]
```
All market skills are returned with hardcoded frequency=1, losing actual market demand data from vacancies.

### ISSUE 6: `save_to_analysis_results` doesn't pass direction_id/discipline_id
**File:** `src/pipeline/db_writer.py:130-136`
```python
async def save_to_analysis_results(run_id, analysis_type, data):
    await pool.execute(
        """INSERT INTO analysis_results (pipeline_run_id, analysis_type, data)
           VALUES ($1, $2, $3)""", ...)
```
The `analysis_results` table has `direction_id`, `discipline_id`, `competency_id` columns, but they're never populated. The `competency_tree` endpoint relies on these for coverage joins.

### ISSUE 7: `DISCIPLINE_ALIASES` is incomplete
**File:** `src/loaders/rpd_loader.py:360-369`
Only 7 English→Russian mappings are defined. Missing:
- `Python для научных вычислений и машинного обучения` (no English variant in aliases)

This isn't a bug (the Russian-only PDFs will be used directly), but it means the parser won't deduplicate if an English variant exists with a different naming pattern.
